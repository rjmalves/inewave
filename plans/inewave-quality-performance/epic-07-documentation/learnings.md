# Epic 07 Learnings: Documentation

## Patterns Established

- **Repo-root Markdown for migration guides**: User-facing migration documentation lives at `/MIGRATION.md` as a standalone Markdown file, not part of the sphinx build. This mirrors the convention already established by `README.md` and `CHANGELOG.md` at the repo root. See `/home/rogerio/git/inewave/MIGRATION.md`.
- **Audience-based language split**: User-facing documentation (`MIGRATION.md`, `CHANGELOG.md`, `docs/`) is written in Portuguese to match the project's existing conventions. Developer-facing infrastructure documentation (`benchmarks/README.md`) is written in English, matching the existing benchmark artifact language. See `/home/rogerio/git/inewave/MIGRATION.md` vs `/home/rogerio/git/inewave/benchmarks/README.md`.
- **CHANGELOG prepend format**: New version entries are added at the top of `CHANGELOG.md` with a `# X.Y.Z` heading followed by `- ` bullet lines (no sub-headings). The entry for v1.13.0 covers 11 bullet points, each one sentence, written in Portuguese. See `/home/rogerio/git/inewave/CHANGELOG.md` lines 1-14.
- **Curated key-numbers section in benchmark README**: `benchmarks/README.md` provides a "Key Performance Characteristics" section with a Markdown table summarizing representative read times and prose for import and aggregation. The table does NOT copy-paste raw results but synthesises the headline numbers, with a pointer to `benchmarks/benchmark_results.md` for full details. See `/home/rogerio/git/inewave/benchmarks/README.md` lines 108-154.
- **Benchmark README reliabilty caveats**: A dedicated "Notes on Result Reliability" section documents known measurement artefacts (Delta noise on low iterations, `__monta_tabela` invisibility with mock single-block data, tracemalloc Python-only scope, import cache warm state). This prevents future developers from misreading Delta regressions. See `/home/rogerio/git/inewave/benchmarks/README.md` lines 183-209.

## Architectural Decisions

- **ticket-032 merged into ticket-031**: The original epic planned a separate ticket for updating sphinx autodoc pages and docstrings. Codebase investigation at refinement time revealed that all RST reference pages use `.. autoclass::` with `:members:` and auto-generate from existing (already-accurate) docstrings. No RST content changes were needed. The only substantive documentation work was the migration guide, changelog, tutorial update, and example update — all naturally scoped into ticket-031. Decision record preserved in `/home/rogerio/git/inewave/plans/inewave-quality-performance/epic-07-documentation/ticket-032-update-api-documentation.md`.
- **MIGRATION.md not added to sphinx toctree**: The migration guide is a standalone repo-root document, not part of the sphinx build. Adding it to the toctree would require RST conversion and ongoing sync. The decision keeps the guide in the same discovery location as `CHANGELOG.md` for tools like GitHub, PyPI, and automated dependency scanners.
- **No benchmarks sphinx page**: `benchmarks/README.md` is the sole benchmark documentation artifact and is not referenced from the sphinx docs index. The `docs/source/index.rst` toctree was not modified. Rationale: benchmarks are a developer/contributor concern, not an end-user API concern. Cross-reference from `MIGRATION.md` section 5 is sufficient discoverability.
- **`set_version()` preserved with backward-compat note**: The tutorial and MIGRATION.md document `read(version=...)` as the preferred idiom while explicitly stating that `set_version()` still works. No deprecation timeline was specified for `set_version()`, consistent with the project's conservative API stability policy. See `/home/rogerio/git/inewave/docs/source/geral/tutorial.rst` lines 125-153 and `/home/rogerio/git/inewave/MIGRATION.md` section 2.1.

## Conventions Adopted

- **MIGRATION.md section ordering**: Dependencies first (breaking upgrade action required), then non-breaking API additions, then deprecations, then internal changes (no user action), then performance cross-reference, then testing numbers. This ordering prioritises what users must act on immediately. See `/home/rogerio/git/inewave/MIGRATION.md` heading structure.
- **Version key documentation with table**: MIGRATION.md section 2.2 provides a table mapping each versioned class name, its module, and its valid `VERSIONS` keys. This prevents the common mistake of passing `version=27` (integer) or `version="v27"` (prefixed string) instead of `version="27"`. See `/home/rogerio/git/inewave/MIGRATION.md` lines 51-63.
- **"Do not edit by hand" label on generated files**: `benchmarks/README.md` Files table explicitly labels `benchmark_results.md` and `profile_report.md` as "Do not edit by hand." This signals to contributors that these files are overwritten by the benchmark runner and hand-edits are lost. See `/home/rogerio/git/inewave/benchmarks/README.md` lines 19-20.
- **Benchmark run commands always specify repo root requirement**: Every "How to Run" command in `benchmarks/README.md` is preceded by an explicit note that commands must be run from the repository root. See `/home/rogerio/git/inewave/benchmarks/README.md` lines 27-28.

## Surprises and Deviations

- **Sphinx autodoc pages required no changes**: The original plan budgeted a full ticket (ticket-032) for updating docstrings and RST API reference pages. Investigation revealed that earlier epics had already kept docstrings accurate as a side effect of implementing the new classes. Zero RST content changes were needed. This saved approximately 2 points of effort. See ticket-032 disposition in `/home/rogerio/git/inewave/plans/inewave-quality-performance/epic-07-documentation/ticket-032-update-api-documentation.md`.
- **`plot_versionamento.py` example kept old-style error demonstration**: The existing example file includes a section showing what happens when a versioned file is read without specifying `version=` (incorrect result, not an exception). Rather than removing this pedagogically useful demonstration, the new API example was appended as a third code block. The `set_version()` call at line 39 was replaced with the `version=` kwarg inline. See `/home/rogerio/git/inewave/examples/plot_versionamento.py`.
- **MIGRATION.md section 4.3 documents `IO[Any]` convention**: The ticket specification included this as an internal change for advanced users. It was retained in the migration guide because library consumers who subclass inewave handler classes or extend `cfinterface.Block` need to know the correct annotation to use when overriding `read`/`write`. This is a documentation scope expansion beyond the original spec, but bounded to one subsection.
- **`AvlCortesFpha` listed in MIGRATION.md section 2.2 VERSIONS table**: The ticket spec listed five versioned classes (Cmarg, Cmargmed, AvlCortesfphaNwv, EcoFpha, AvlDesvfphaS). The actual implemented VERSIONS table in the migration guide lists Cmarg, Cmargmed, Pivarm, Pivarmincr, and AvlCortesFpha — reflecting the actual state of the codebase after the version audit (ticket-014) rather than the pre-audit spec. See `/home/rogerio/git/inewave/MIGRATION.md` lines 54-63.

## Recommendations for Future Epics

- **Write migration guides before version bumps, not after**: The MIGRATION.md for v1.13.0 was written at the end of the upgrade plan. Future major releases should draft the migration guide incrementally during the epic sequence, updating it as each epic adds user-visible changes. The structure established in `/home/rogerio/git/inewave/MIGRATION.md` can serve as a template for v1.14.0+.
- **Budget a documentation ticket at the end of every multi-epic plan**: This epic took 3 points and produced 22 KB of documentation covering 7 epics of change. The effort is non-negligible and should be explicitly scoped in future plans, not treated as an afterthought. Even for smaller plans, a 1-point documentation ticket for CHANGELOG and tutorial updates pays off in downstream consumer clarity.
- **Benchmark README should be updated when benchmark infrastructure changes**: If `benchmarks/run_benchmarks.py` or `benchmarks/bench_read.py` are modified in future epics, the `benchmarks/README.md` "How to Run" and "Architecture" sections require corresponding updates. The current README accurately describes the v1.13.0 benchmark architecture. See `/home/rogerio/git/inewave/benchmarks/README.md`.
- **Version key table in MIGRATION.md should be kept in sync with VERSIONS dicts**: If future epics add new versioned classes or new version keys to existing VERSIONS dicts, the table in `MIGRATION.md` section 2.2 must be updated in the same PR. The table is currently the only place where all versioned classes are listed in a single view.
